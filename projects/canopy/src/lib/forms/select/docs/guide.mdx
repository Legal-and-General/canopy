# Select

<p class="standfirst">A drop down list that enables the user to choose between several options.</p>

![Select hero](docs/select/hero.png)

## Usage

Selects should only be used when space constraints or the number of options make **Radio buttons**, **Segmented controls**, or **Checkboxes** impractical. These alternative inputs typically provide a superior experience by keeping options visible and scannable.

#### When to use

- **More than 5 options:** Use a select only when there are more than five options. For five or fewer, use radio buttons to reduce the interaction cost.
- **Unpredictable data:** Use a select when options cannot be easily typed. For data users know by heart (like Date of Birth or Postcodes), use a text input instead.

#### When to avoid

- Do not use a select for binary (Yes/No) choices.
- Avoid using a select if the list is extremely long (e.g., over 20 items); consider a combobox with autocomplete instead.

## How it works

The select component must always be used with a visible label. You should also include a hint if the user needs additional context to make the correct selection.

Do not pre-select an option by default. The only exception is when pre-populating a form with values the user has previously selected. Use a neutral placeholder, such as "Select an option", to force an intentional choice and prevent accidental submissions.

While the input field follows Canopy styling, the dropdown menu itself is rendered by the browser or operating system. This ensures the menu is optimised for the user's specific device – such as using the native picker on iOS or Android – and maintains high levels of accessibility.

The component must support standard keyboard navigation: arrow keys to navigate, **Enter** or **Space** to select, and **Esc** to close. It should also support "type-ahead", allowing users to jump to an option by typing its first letter.

List options in a predictable order – such as alphabetical, chronological, or by frequency of use – to help users scan the list quickly. Like all form inputs, the width of the select must match the maximum expected length of the content to provide a visual cue about the data required.

#### Grouping

The `optgroup` element must be used to semantically group related options within a select menu, particularly when lists are long enough to cause high cognitive load. It provides a visual and programmatic hierarchy that helps users skim categories, but `optgroup` should never be used for nesting as HTML only supports a single level of grouping. While `optgroup` improves organisation, screen reader support for group labels can be inconsistent; therefore, option labels should remain descriptive enough to be understood out of context, or supplemented with `aria-label` if the group context is critical for selection.

## Do and don't

### Do

![Select do](docs/select/do.png)

1. **Do** use a visible label and, where necessary, a hint to provide context.
2. **Do** use a select when there are more than five options.
3. **Do** use a neutral placeholder like "Select an option" to ensure the user makes an intentional choice.
4. **Do** order options logically – such as alphabetically or by frequency of use – to aid scannability.
5. **Do** match the width of the select to the expected length of the options.

### Don't

![Select dont](docs/select/dont.png)

1. **Don't** use a select for five or fewer options; use Radio buttons or Segmented controls instead.
2. **Don't** use a select for binary choices (e.g. Yes/No).
3. **Don't** use a select for data users know by heart, such as Date of Birth; use a Text input instead.
4. **Don't** use a select if the list is excessively long (e.g. over 20 items); consider a Combobox with autocomplete.
5. **Don't** pre-select an option by default unless you are pre-populating the form with the user's previously saved data.
6. **Don't** use disabled options within the list; if an option is unavailable. It is better to remove it or explain why it cannot be selected via an error message.

## Developer reference

### LgSelectDirective inputs

| Name   | Description                                         | Type     | Default                       | Required |
|--------|-----------------------------------------------------|----------|-------------------------------|----------|
| `id`   | HTML ID attribute, auto generated if not provided   | `string` | `lg-select-${nextUniqueId++}` | No       |
| `name` | HTML Name attribute, auto generated if not provided | `string` | `lg-select-${nextUniqueId++}` | No       |

### LgSelectFieldComponent inputs

| Name    | Description                                                                                                            | Type      | Default                       | Required |
|---------|------------------------------------------------------------------------------------------------------------------------|-----------|-------------------------------|----------|
| `id`    | HTML ID attribute, auto generated if not provided. This will also propagate to the input `id` and form `for` attribute | `string`  | `lg-select-${nextUniqueId++}` | No       |
| `block` | Property to make the select field full width (for small screens only)                                                  | `boolean` | `false`                       | No       |
